# Action Guide — 工作流指南

> 每次开始工作前必读本文件，按步骤执行。

---

## 标准工作流（每个 Session 必须遵循）

### Step 1: 查看架构文档
```
打开 architect.md
```
- 确认当前要做的模块在架构中的位置
- 确认相关超参数、接口定义

### Step 2: 查看开发计划
```
打开 plan.md
```
- 确认当前 Phase 和具体任务编号（如 P2-3）
- 确认任务依赖是否已满足

### Step 3: 查看状态文档
```
打开 status.md
```
- 确认"下一步行动"列表
- 确认是否有阻塞项需要先解决

### Step 4: 执行任务
- 按 plan.md 中的任务顺序执行
- 每个子任务完成后立即更新文档（不要积累）

### Step 5: 完成后更新文档

#### 更新 plan.md
将已完成任务的 `- [ ]` 改为 `- [x]`：
```markdown
- [x] P2-1 实现 ScaledDotProductAttention
```

#### 更新 status.md
1. 修改"最后更新"日期
2. 更新"当前阶段"
3. 更新"总体进度"表格中对应 Phase 状态
4. 更新 Phase 详情表格（将完成的任务标为"已完成"）
5. 更新"下一步行动"为下一个未完成任务

---
#### 更新 architect.md
1. 针对已完成任务有与原architect不一致的地方，进行调整修改


## 任务状态标记规范

| 标记 | 含义 |
|---|---|
| 未开始 | 尚未启动 |
| 进行中 | 当前正在做 |
| 已完成 | 验证通过，可以进入下一步 |
| 阻塞 | 有依赖未满足或遇到问题 |

---

## 文件职责速查

| 文件 | 职责 | 修改频率 |
|---|---|---|
| `architect.md` | 架构设计、模块定义、超参数 | 极少（设计变更时） |
| `plan.md` | 任务清单、依赖关系 | 每完成一个任务更新一次 |
| `status.md` | 当前进度、阻塞、下一步 | 每个 Session 至少更新一次 |
| `action_guide.md` | 工作流程本身 | 极少（流程变更时） |

---

## 快速检查清单（Session 结束前）

- [ ] plan.md 中已完成任务已打勾
- [ ] status.md 中"最后更新"日期已更新
- [ ] status.md 中"下一步行动"已更新为下一个任务
- [ ] 新增代码文件已保存在正确目录
- [ ] 如有阻塞，已记录在 status.md 的"当前阻塞"中

## 本地 vs 云端切换提醒

**本地开发时：**
- 使用 `configs/base.yaml`
- 数据使用 `data/processed/debug_10k/`（小子集）
- 设备：`cpu` 或 `mps`

**切换到云端训练时：**
- 使用 `configs/cloud.yaml`
- 数据使用完整数据集
- 设备：`cuda`
- 确认 Phase 3 全部完成后再切换
